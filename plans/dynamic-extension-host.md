# Dynamic Built-in Extension Host Plan

## Context

- Built-in extensions are currently organized under `extensions/builtin/*`, but they are not fully self-contained at runtime.
- Renderer built-ins are statically imported by `src/extensions/registry.tsx` via `getNativeExtensions()`.
- Bundled manifests are statically imported by `electron/extensionService.ts` through `getBundledExtensionManifests()`.
- Main-process Git commands are wired directly in `electron/main.ts` and exposed as hard-coded `window.stackdock.git.*` methods in `electron/preload.ts`.
- Core workspace UI owns Git state/actions in `src/components/workspace/WorkspaceShell.tsx` and passes them to `extensions/builtin/git/renderer/index.tsx` through `WorkspaceExtensionContext.git` and `gitActions`.
- `extensions/mainRegistry.ts` also statically imports the Pi built-in for terminal profiles/integration.
- Local, non-native extensions already use iframe contributions through `ExtensionFrame`, but trusted built-ins use direct React modules and main-process code.

## Approach

Implement a trusted bundled extension host in phases:

1. Add a generic extension runtime contract for bundled main and renderer modules.
2. Load bundled manifests from extension folders instead of hard-coded imports.
3. Replace Git-specific IPC/preload APIs with a namespaced generic extension RPC.
4. Move Git main IPC registration into `extensions/builtin/git/main/index.ts`.
5. Move Git state/actions out of `WorkspaceShell.tsx` and into the Git renderer module.
6. Replace static native renderer registration with a dynamic built-in renderer module loader.
7. Move Pi terminal-profile/integration registration into a bundled main extension activation hook.
8. Remove obsolete core Git fields once dashboard/explorer dependencies have replacement APIs.

Important scope decision: dynamic **main-process** code is for trusted bundled extensions only in this plan. Local/user-installed extensions remain sandboxed iframe extensions unless a later security review adds third-party main extension permissions.

## Reuse

- Reuse existing manifest contribution types in `src/shared/types.ts`: `ExtensionManifest`, `ExtensionViewContribution`, `ExtensionStatusBarContribution`, and `ExtensionTerminalCommandHookContribution`.
- Reuse `ExtensionFrame` for local iframe extensions; do not replace it.
- Reuse existing enablement logic in `src/extensions/enablement.ts`.
- Reuse Git command implementation in `extensions/builtin/git/main/gitService.ts`; only move registration, not command internals.
- Reuse existing `NativeExtension` interface as the renderer-module shape initially, then extend only where required.
- Reuse existing workspace actions in `WorkspaceExtensionContext.actions` for generic UI operations like `openFile`, `revealFolder`, `createSession`, and `openView`.
- Reuse validation helpers in `electron/validation.ts` for main extension RPC arguments.

## Assumptions

- Bundled extensions are trusted and may run in the Electron main process.
- Local/user-installed extensions are not allowed to register arbitrary main-process code in this implementation.
- A small static loader map generated by Vite `import.meta.glob` is acceptable for renderer bundling, because extension IDs/paths are not hand-maintained in source code.
- Git's Source Control activity view can be visible even outside a repository; the panel already supports displaying a "Not a git repo." state. This avoids keeping core Git repo detection solely for `when: 'gitRepo'`.
- Dashboard and Explorer can call a generic extension RPC (`stackdock.git/status`, `stackdock.git/ignored`) until they are fully extension-owned.
- Existing iframe local extensions continue to receive a limited bridge; Git-specific iframe messages should be deprecated but not immediately removed until tests confirm no bundled/local extension uses them.

## Steps

### Phase 1 — Define extension runtime contracts

1. In `src/shared/types.ts`, extend `ExtensionManifest` with optional runtime entry fields:
   - `main?: string`
   - `renderer?: string`
   - Keep existing `contributes` unchanged.
   - Expected result: manifests can point to extension-owned main/renderer modules.
2. In `src/extensions/runtimeTypes.ts`, define `ExtensionInvoke`, `ExtensionRendererContext`, and `ExtensionMainRpc` shared renderer-facing types.
   - Include `invoke(command: string, ...args: unknown[]): Promise<unknown>` scoped to the current extension ID.
   - Include generic actions copied from `WorkspaceExtensionContext.actions` that are not Git-specific.
   - Expected result: renderer extensions do not need `window.stackdock.git` or `ctx.gitActions`.
3. In `src/extensions/extensionTypes.ts`, add optional `activateRenderer?(ctx: ExtensionRendererContext): NativeExtension` support or define that dynamically loaded renderer modules default-export `NativeExtension`.
   - Keep existing `NativeExtension` shape for compatibility.
   - Expected result: existing renderer components can migrate without a large UI rewrite.
4. In `electron/extensionHost.ts`, define `ExtensionMainContext` with:
   - `extensionId`
   - `manifest`
   - `subscriptions`
   - `rpc.handle(command, handler)`
   - `workspaceTrust.requireTrusted(reason)` or a callback to existing trust logic if available in main
   - access to safe core services needed by built-ins (`terminal`, `settings`, `workspaces` only where already supported)
   - Expected result: main extensions register handlers through one scoped API.

### Phase 2 — Load bundled manifests from extension folders

5. Add `stackdock.extension.json` to each bundled extension folder under `extensions/builtin/*`.
   - Copy fields from the corresponding current `manifest.ts` files.
   - Add `renderer: "./renderer/index"` for React/native renderer built-ins.
   - Add `main: "./main/index"` for Git and Pi only at first.
   - For Git, change view `when` from `gitRepo` to `always`; keep status bar `when` unset or `always` and let the renderer return `null` when not in a repo.
   - Expected result: bundled extension metadata lives beside each extension.
6. In `electron/extensionService.ts`, replace `getBundledExtensionManifests()` hard-coded imports with a bundled-directory scanner.
   - Add `getBundledExtensionRoots()` that resolves `extensions/builtin/*` in dev and the copied bundled extension directory in packaged builds.
   - Reuse `normalizeManifest(raw, root)` for bundled manifests with a new `source: 'bundled'` mode.
   - Validate `main` and `renderer` entries with existing `validateEntry` logic.
   - Expected result: `api.extensions.list()` returns bundled manifests without importing every manifest module.
7. Preserve duplicate-ID protection between bundled and local extensions in `loadExtensions(settings)`.
   - Expected result: local extensions cannot shadow bundled extension IDs.
8. Keep old `manifest.ts` files temporarily or convert them to import the JSON, but stop using them for runtime discovery.
   - Expected result: minimal churn for tests and incremental migration.

### Phase 3 — Add generic extension RPC bridge

9. In `electron/extensionHost.ts`, implement an `ExtensionHost` class with:
   - `activateBundledExtensions(manifests)`
   - `registerRpcHandler(extensionId, command, handler)`
   - `invoke(extensionId, command, args)`
   - lifecycle cleanup for subscriptions and duplicate handler errors.
   - Expected result: main extension code can register `status`, `stage`, `discard`, etc. without direct `ipcMain.handle('git:*')` calls.
10. In `electron/main.ts`, instantiate the host after settings/extensions load and before windows need extension APIs.
    - Call `await extensionHost.activateBundledExtensions(await ensureExtensionsLoaded(settings))`.
    - Add one IPC handler: `ipcMain.handle('extensions:invoke', (_event, extensionId, command, args) => extensionHost.invoke(...))`.
    - Expected result: `electron/main.ts` contains no Git service imports or `git:*` handler registrations.
11. In `electron/preload.ts`, add:
    - `extensions.invoke(extensionId: string, command: string, args?: unknown[]): Promise<unknown>`.
    - Validate argument shapes before sending to IPC: extension ID non-empty string, command non-empty string, args array defaulting to `[]`.
    - Expected result: renderer code can call extension main handlers generically.
12. In `src/shared/types.ts`, add the generic invoke signature to `StackDockApi.extensions`.
    - Keep `StackDockApi.git` temporarily for compatibility until Phase 7 removes it.
    - Expected result: TypeScript consumers can use `api.extensions.invoke`.
13. Add `tests/extensionHost.test.ts` covering:
    - duplicate command registration fails for one extension
    - same command name is allowed under different extension IDs
    - unknown extension/command returns a clear error
    - activation cleanup removes handlers
    - Expected result: generic RPC is verifiable without launching Electron.

### Phase 4 — Move Git main registration into the Git extension

14. Create `extensions/builtin/git/main/index.ts` exporting `activateMain(ctx: ExtensionMainContext)`.
    - Register commands: `status`, `branches`, `diff`, `fileContents`, `stage`, `unstage`, `discard`, `ignore`, `commit`, `addAll`, `switchBranch`, `push`, `pull`, `pullMerge`, `abortMerge`, `fetch`, `ignored`.
    - Each handler must call the same validation currently in `electron/main.ts`: `assertAbsolutePath`, `assertNonEmptyString`, arrays mapped through `assertNonEmptyString`.
    - Expected result: Git main process API is declared under `extensions/builtin/git`.
15. In `electron/main.ts`, delete the Git service import and all `ipcMain.handle('git:*')` registrations.
    - Expected result: no `../extensions/builtin/git/main/gitService` import remains in `electron/main.ts`.
16. In `electron/preload.ts`, leave `window.stackdock.git.*` as a compatibility shim that calls `extensions.invoke('stackdock.git', command, args)`.
    - Mark each shim with a TODO/deprecation comment.
    - Expected result: existing renderer code still works while migration continues.
17. Update `tests/gitService.test.ts` only if imports need to move; prefer keeping service-unit tests pointed at `gitService.ts`.
    - Expected result: Git command internals remain independently tested.

### Phase 5 — Dynamic renderer module loader

18. Replace `src/extensions/registry.tsx` hard-coded imports with a dynamic loader in `src/extensions/nativeLoader.tsx`.
    - Use `import.meta.glob('../../extensions/builtin/*/renderer/index.tsx')`.
    - Build a map from normalized manifest `renderer` path to loader function.
    - Export `loadNativeExtension(manifest): Promise<NativeExtension | null>`.
    - Expected result: adding a bundled renderer extension requires a manifest/entry file, not editing `registry.tsx`.
19. Change `src/extensions/registry.tsx` to only re-export enablement helpers or delete it after callers migrate.
    - Expected result: no static imports of `explorerExtension`, `gitExtension`, `sessionsExtension`, `headlessExtension`, `workspaceStatusExtension`, or `piExtension` remain.
20. In `src/extensions/ExtensionProvider.tsx`, replace the synchronous `nativeExtensions` initialization with async loading after `api.extensions.list()`.
    - State should include `nativeExtensions: Map<string, NativeExtension>` and `nativeLoadErrors` if needed.
    - For each bundled manifest with `renderer`, call `loadNativeExtension(manifest)`.
    - Keep iframe local extensions unchanged.
    - Expected result: views render after manifest + module loading completes.
21. Add loading/error fallback UI for a native extension whose module fails to load.
    - Expected result: one broken extension does not break the entire workspace.

### Phase 6 — Move Git renderer state/actions out of WorkspaceShell

22. In `extensions/builtin/git/renderer/index.tsx`, create a Git-specific React container component, e.g. `GitExtensionView`.
    - Move state currently owned by `WorkspaceShell.tsx`: status, selected file, staged/unstaged selected paths, git error, last selected path.
    - Keep `GitPanel` presentational if possible.
    - Expected result: Git view owns its own state.
23. In the Git renderer container, replace `api.git.*` calls with `ctx.invoke(command, args)` or `api.extensions.invoke('stackdock.git', command, args)`.
    - Example: `ctx.invoke('status', workspace.path)` instead of `api.git.status(workspace.path)`.
    - Expected result: Git renderer talks only to Git extension main RPC.
24. Move Git diff-opening behavior out of `WorkspaceShell.tsx` into Git renderer using existing generic editor actions.
    - If existing generic actions are insufficient, add a generic `actions.openDiff({ path, original, modified, staged, untracked })` to `WorkspaceExtensionContext` rather than Git-specific actions.
    - Expected result: selecting a Git file can still open the diff editor without core owning Git state.
25. Move Git polling interval and refresh interval setting into the Git renderer container.
    - Read extension config from `ctx.settings?.extensions.config['stackdock.git']` or a helper.
    - Expected result: Source Control refresh cadence remains configurable.
26. Move Git command palette commands from `gitExtension.getCommands` to use Git renderer-owned state and generic invoke.
    - If command generation currently requires synchronous state from the active Git view, add an extension command registry in renderer context that the Git container updates when status changes.
    - Expected result: `Git: Stage All`, `Git: Commit`, `Git: Pull`, etc. still appear and work.
27. In `src/components/workspace/WorkspaceShell.tsx`, remove Git action functions after the Git container owns them:
    - `refreshGit`
    - `loadEditorDiff` if replaced
    - `openGitDiff` if replaced
    - `selectGitFile`
    - `stage`, `stagePaths`, `stageAll`
    - `unstage`, `unstagePaths`
    - `discardPath`, `discard`, `discardPaths`
    - `ignore`
    - `commit`, `commitStaged`, `stageAllAndCommit`
    - `switchGitBranch`, `runGitRemoteAction`, `abortGitMerge`
    - Expected result: `WorkspaceShell.tsx` no longer contains Source Control business logic.
28. In `src/extensions/extensionTypes.ts`, remove `git`, `isRepo`, `gitActions`, and `actions.refreshGit` from `WorkspaceExtensionContext` only after all references are gone.
    - Expected result: extension context is generic.

### Phase 7 — Replace remaining core/preload Git dependencies

29. Update `src/components/dashboard/WorkspaceDashboard.tsx` to call `api.extensions.invoke('stackdock.git', 'status', [workspace.path])` instead of `api.git.status`.
    - Expected result: dashboard dirty counts still display.
30. Update `extensions/builtin/explorer/renderer/FileTree.tsx` to call `api.extensions.invoke('stackdock.git', 'ignored', [rootPath, paths])` or use a new generic `ctx.invokeExtension('stackdock.git', 'ignored', ...)` helper.
    - Expected result: file tree ignore filtering still works.
31. Remove `StackDockApi.git` from `src/shared/types.ts` after all `api.git` references are gone.
    - Expected result: no hard-coded Git preload API remains in TypeScript.
32. Remove the compatibility `git` object from `electron/preload.ts`.
    - Expected result: renderer has only generic extension RPC.
33. Remove Git-specific iframe bridge cases from `src/extensions/ExtensionFrame.tsx`:
    - `git` field in `stackdock.getContext`
    - `stackdock.refreshGit`
    - Keep generic workspace/session fields and actions.
    - Expected result: local iframe bridge is no longer Git-aware.

### Phase 8 — Move Pi main registration into extension activation

34. Create `extensions/builtin/pi/main/index.ts` exporting `activateMain(ctx: ExtensionMainContext)`.
    - Register terminal profile contribution currently returned by `getBundledTerminalProfiles()`.
    - Register terminal integration currently created by `createPiTerminalIntegration(settings)`.
    - Expected result: Pi main functionality lives under the Pi extension.
35. Extend `ExtensionMainContext` with `terminal.registerProfile(profile)` and `terminal.registerCommandIntegration(integration)`.
    - Expected result: terminal-related built-ins do not need `extensions/mainRegistry.ts` hard-coded imports.
36. Refactor `extensions/mainRegistry.ts` to read registered profiles/integrations from `ExtensionHost` instead of importing Pi directly.
    - Expected result: no static Pi import remains in main registry.
37. Ensure extension enablement is respected for registered terminal profiles/integrations.
    - Expected result: disabling the Pi extension disables its terminal profile/integration.

### Phase 9 — Build and packaging support

38. In `electron/tsconfig.json`, include `../extensions/**/*.ts` needed by main extension modules, but exclude renderer TSX from the Electron main build if Vite owns it.
    - Expected result: `extensions/builtin/*/main/*.ts` compiles into `dist-electron/extensions/...`.
39. Add `scripts/copy-extension-assets.cjs` to copy `extensions/builtin/**/stackdock.extension.json` and any non-TS assets required at runtime into the packaged output.
    - Expected result: packaged apps can scan bundled extension manifests.
40. Update `scripts/build-if-needed.cjs` and `package.json` build scripts to run the extension asset copy after TypeScript/Vite build.
    - Expected result: `npm run build` produces runtime extension files.
41. Update `package.json` `build.files` to include copied bundled extension manifests/main modules if they are not already under `dist-electron/**/*`.
    - Expected result: electron-builder packages the dynamic extension host correctly.
42. Verify source maps/paths in dev and packaged mode.
    - Expected result: `npm run dev`, `npm run build`, and packaged app all resolve extension entries the same way through helper functions.

### Phase 10 — Cleanup old manifest modules and static references

43. Delete or deprecate `extensions/builtin/*/manifest.ts` once JSON manifests are the source of truth.
    - Expected result: no built-in manifest is maintained in two places.
44. Delete `src/extensions/registry.tsx` if only enablement re-exports remain; import enablement from `src/extensions/enablement.ts` directly.
    - Expected result: no static built-in renderer registry remains.
45. Search for and remove all remaining hard-coded references:
    - `../extensions/builtin/git` from `electron/main.ts`
    - `git:` IPC channels
    - `window.stackdock.git`
    - `api.git`
    - `gitActions`
    - static `explorerExtension`, `gitExtension`, `sessionsExtension`, `headlessExtension`, `workspaceStatusExtension`, `piExtension` imports
    - Expected result: built-ins are discovered from manifests and loaded through extension host paths.

## Risks

- **Security risk:** generic extension RPC can become an arbitrary privileged bridge. Mitigation: only bundled extensions can register main handlers in this plan; validate all arguments in each handler; namespace all commands by extension ID.
- **Packaging risk:** dynamic main modules/manifests may work in dev but be missing in packaged builds. Mitigation: add copy script, package file entries, and packaged-app manual verification.
- **Renderer bundling risk:** Vite cannot truly import arbitrary runtime files. Mitigation: use `import.meta.glob` for bundled renderer modules; local extensions remain iframe based.
- **State regression risk:** moving Git state out of `WorkspaceShell.tsx` can break diff tabs, command palette commands, polling, and toast/error behavior. Mitigation: migrate Git renderer state before deleting core state, and keep compatibility shims during transition.
- **Enablement regression risk:** terminal integrations and status bar contributions may ignore disabled extensions if registration is global. Mitigation: extension host must re-evaluate enablement and dispose registrations when settings reload.
- **API compatibility risk:** local iframe extensions may rely on `git` in `stackdock.getContext` or `stackdock.refreshGit`. Mitigation: deprecate first, search examples/docs, then remove after a compatibility window if needed.
- **Circular dependency risk:** shared runtime types imported by both Electron and renderer can accidentally import React or Electron-only modules. Mitigation: keep `src/shared/types.ts` pure data types and keep React-specific runtime types in `src/extensions/runtimeTypes.ts`.
- **Rollback:** if a phase breaks app startup, revert that phase only. The phased compatibility shims (`window.stackdock.git` and old context fields) allow stopping after Phases 3-5 without forcing the full Git state migration immediately.
